### Durable terminal recording and replay architecture

The full design for durable PTY-output recording, durable replay across the in-memory ring's window, closed-session replay, and backend-restart display-history recovery lives in [`docs/terminal-recording.md`](../terminal-recording.md). That document is normative for any future recording slice. The bullets below are the load-bearing invariants that anchor the design — implementation slices MUST preserve every one of them, and any drift goes through `docs/terminal-recording.md` first, not through code.

**Status (this slice).** Design landed (step 1a), typed-config foundation landed (step 1b), schema + repository foundation landed (step 2), the orchestrator's recording-writer foundation landed (step 3 — partial), the durable replay read API foundation landed (step 4 — partial), the frontend replay viewer foundation landed (step 5 — partial), AND backend startup reconciliation has now landed in full (step 6 — both the `terminal_sessions` + `session_events` close AND the closed-recording-marker write). The startup slice runs a single repository pass at boot — AFTER the database pool is ready and BEFORE the listener binds — that locks every `terminal_sessions WHERE status IN ('starting', 'active', 'detached')` row, transitions each to `closed`, appends one matching `session_events { kind: closed, payload: { reason: "startup_reconciliation", previous_status, reconciled_at } }` row per session, AND for any reconciled session that has at least one `terminal_recording_chunks` row appends a single `terminal_recording_markers { kind: closed, seq: MAX(seq_end), payload: { reason: "startup_reconciliation", previous_status, reconciled_at } }` row inside the SAME database transaction (skipping the marker insert when an equivalent `(closed, seq=MAX(seq_end))` marker already exists). The pass writes ZERO `audit_events` rows on either branch (matches the existing close-path audit shape; if a future slice decides restart-reconciliation IS audit-worthy, it adds a dedicated `terminal_session_reconciled` audit kind through the standard migration pattern). It does NOT delete any `terminal_sessions` / chunk / marker row. Chunk rows remain untouched and the existing closed-session replay path keeps working. The closed marker carries public metadata only — never PTY bytes, never `client_info`, never peer banners — and is built field-by-field. The pass is idempotent on three axes: (a) a second boot finds no orphans and writes nothing; (b) the marker insert uses `ON CONFLICT DO NOTHING` against the partial unique index `terminal_recording_markers_session_closed_seq_uidx` on `(terminal_session_id, seq) WHERE kind = 'closed'`, so a partial earlier run, an operator-written marker, or a racing writer all collapse to a single row at the schema layer; (c) sessions without chunks never reach the marker branch. A failure to reconcile fails startup before the listener binds; the operator sees the boot abort instead of a backend that serves stale "live" rows the UI cannot resume. Live PTY recovery across the restart is intentionally NOT in scope (the `russh::Channel` is unrecoverable; recording captures display history, not the live shell). The frontend slice adds a read-only replay viewer (`apps/web/src/lib/app/views/RecordingReplayView.svelte`) reachable from the Sessions list via a new `View recording` action. The viewer fetches metadata, pages chunks page-by-page, and writes decoded bytes directly into a read-only `XtermRenderer` — keyboard input is disabled (`xtermOnly: { disableStdin: true }`) and there is no input listener, no live SSH client, and no WebSocket attach. Decoded chunk bytes are NEVER stashed in `$state`, `localStorage`, or `sessionStorage`; the viewer holds them only long enough to call `renderer.write(...)` per chunk. Markers are rendered as a metadata strip only (key/value preview), never as terminal output. The Sessions list deliberately does NOT pre-fetch metadata for every row (would be N+1); the affordance is offered for any non-`starting` row and the viewer's metadata gate honestly surfaces "No recording available" when an opened session turns out empty. Three owner-scoped GET routes — `/api/v1/terminal-sessions/:id/recording/{metadata,chunks,markers}` — surface the persisted recording artefacts. Chunk payloads cross the wire ONLY as base64 (`data_b64`); raw bytes never reach a log line, error path, or audit row. The repository surface is session-scoped (`TerminalRecordingRepository::get_metadata`, `list_chunks`, `list_markers`); owner scoping happens at the route layer via the existing `terminal_sessions.owner_id == user.user_id()` filter, with foreign-owned and missing collapsing to a byte-identical 404. Markers wired in the writer slice are `started`, `closed`, `resized`, and `replay_gap`; `attached` / `detached` / `reattached` are deferred to a follow-up that threads the writer handle through the attach surface. Supported runtime mode is `terminal_recording.enabled = true` AND `encryption.mode = disabled` (plaintext-at-rest) only — the production-required `encryption.mode = required` writer has not landed and the backend FAILS TO BOOT in that combination so an operator cannot silently degrade to plaintext. Recording is OFF by default; flipping it on AND choosing `encryption.mode = disabled` (dev only — operator accepts plaintext-at-rest) is what activates the writer. There is still no live attach fallback to durable chunks. The retention slice's startup-only sweep (Stage A — step 8a) has now landed: the backend boot, AFTER terminal-session reconciliation and BEFORE the HTTP listener binds, runs a single bounded retention sweep that calls `TerminalRecordingRepository::list_eligible_for_retention(retention_days, now, cleanup.batch_size)` for the candidate ids and then drives `purge_for_retention` once per session (each in its own transaction). The sweep is gated by `terminal_recording.cleanup.enabled` AND `terminal_recording.cleanup.startup_sweep_enabled`; both default to `true`. It is independent of `terminal_recording.enabled` (12.6 independence rule) so an existing recording corpus continues to age out even when the chunk writer is currently disabled. Failure is fail-soft: a repository error during candidate selection or a per-session purge logs the static category tag `"retention_sweep_failed"` (plus the stage and a coarse `error_kind`) and the boot continues to the listener bind; the first per-session error stops the current sweep and remaining eligible sessions wait for the next boot. Per-session audit rows continue to be written by the existing `purge_for_retention` primitive (`actor_id IS NULL`, invisible to `recent_for_actor`). The Stage B periodic managed worker has now also landed (`relayterm-terminal::retention_worker::spawn_recording_retention_periodic_worker`, wired from `apps/backend/src/main.rs` after `AppState` construction and BEFORE the listener accepts connections). The worker is gated by `cleanup.enabled`, `cleanup.periodic_sweep_enabled`, AND a non-zero `cleanup.sweep_interval_seconds`; the gating helper is `periodic_worker_config_if_enabled`. The first periodic tick fires AFTER `interval` has elapsed (Stage A already drained the eligibility set on boot, so an immediate periodic tick would be a redundant no-op). On every tick the worker takes a Postgres advisory lock — `pg_try_advisory_lock(RECORDING_RETENTION_ADVISORY_LOCK_KEY)` via `relayterm-db::PgRetentionAdvisoryLock` — runs the same `run_recording_retention_sweep` body Stage A uses (cadence-tagged `SweepCadence::Periodic`), and releases the lock; a second backend instance that fails to acquire the lock skips the tick silently (`debug!` only). Shutdown is wired through a `tokio::sync::watch::Receiver<bool>` — the same closure that signals graceful shutdown to `axum::serve` fans the signal out to the worker, and `main` awaits the worker's `JoinHandle` after `axum::serve` returns. The worker is fail-soft: a sweep error or an advisory-lock error logs the static category tag `"retention_sweep_failed"` and the loop continues to the next tick. No encryption / compression implementation, no VT snapshot observer, no export / download endpoint, no admin / cross-user recording access exist yet. Implementation slices 5 (live-attach bridge + viewer polish), 7, 9 (`docs/terminal-recording.md` Section 13) remain ahead.

- **Scope is display-history**, not live-PTY persistence. A future recording subsystem captures the same PTY OUTPUT bytes the live wire already carries and stores them durably so a reconnect after the in-memory ring's window OR a closed-session review can reproduce what the operator's terminal printed. It does NOT resurrect the live `russh::Channel` after a backend restart — long-running tmux/screen-style detached-PTY persistence beyond `DETACHED_LIVE_PTY_TTL` remains separate, deliberate future work owned by a different slice.
- **Recording is OFF by default and config-gated.** A self-hosted operator opts in at the config layer; production-mode boot validation refuses to start with recording enabled and no recording master key. The recording master key is SEPARATE from the SSH-identity vault master key — compromise of one MUST NOT leak the other.
- **Output bytes only; input is NOT recorded in v1.** PTY output frames are persisted as renderer-neutral byte chunks (with `seq_start`/`seq_end`/`byte_len`); resize, attach/detach/reattach, and lifecycle close are persisted as separate metadata MARKERS. Client-to-server `Input` bytes (passwords typed at non-echoing prompts, sudo / vault prompts, confirmed paste content) are NEVER persisted by the recording subsystem in v1. A later opt-in keystroke-audit surface — if it ever lands — is its own slice with its own UI warnings and is NOT a default.
- **Persisted format is renderer-neutral.** Chunks store the exact bytes the live wire's `Output.payload` carries; the replay path re-emits them through the same `Output { seq, data }` wire variant (and the same `RTB1` binary envelope) the live attach uses. No grid model, no shaper state, no per-renderer glyph state is persisted. A future `terminal_vt_snapshots` table is an OPTIMISATION on top of the chunk stream (fast-forward to nearest snapshot, then replay chunks past it), NEVER a replacement.
- **Recording rows live in their own table family** (`terminal_recording_chunks`, `terminal_recording_markers`, future `terminal_vt_snapshots`), separate from `audit_events` and `session_events`. Foreign keys to `terminal_sessions(id)` use `ON DELETE RESTRICT`. Recording rows are NEVER cascade-deleted with their session row; they are removed only by the explicit retention sweep (Section 12 of the design doc).
- **Recording bytes NEVER appear in `audit_events.payload`.** The existing `AUDIT_FORBIDDEN_SUBSTRINGS` rule in `crates/relayterm-api/tests/api.rs` is the backstop and applies in full to recording. Lifecycle audit kinds for recording (e.g. a future `recording_purged`) carry public metadata only — ids, counts, reason codes — never chunk bytes, never marker payload bytes, never error text.
- **Recording bytes NEVER appear in any log, panic message, error response, dashboard summary, session-list cell, `data-*` attribute, frontend `localStorage`/`sessionStorage`, or any other surface that is not the dedicated recording read endpoints.** Chunk repository / writer types redact `payload` in their `Debug` impls to `seq_start..=seq_end + len` only, mirroring the in-memory `OutputFrame::Debug` rule. The `(session_id, last_seen_seq)` `sessionStorage` pointer remains the only durable client-side hint and continues to carry zero bytes.
- **Owner-scoped reads only.** Every recording read goes through `AuthenticatedUser` and joins through `terminal_sessions.owner_id == user.user_id()`. A foreign session id collapses to a byte-identical 404 BEFORE any chunk row is considered. A session with recording disabled (no chunk rows) returns 404 from the metadata endpoint — recording rows are NEVER surfaced through the generic session list / detail / dashboard / activity surfaces.
- **Live replay still uses the in-memory ring first.** The wire variants `ReplayStart` / `Output` / `ReplayEnd` / `ReplayWindowLost` do NOT change. When a bookmark predates the in-memory ring's oldest retained frame, the handler sources older frames from the chunk store, then bridges into the ring at the first ring frame past the chunk stream's `latest_seq`, then resumes live fanout. The `min_live_seq` floor rule still applies.
- **Durable gaps are surfaced, never papered over.** A `replay_gap` recording marker between `from_seq` and `latest_seq` MUST surface as `ReplayWindowLost` to the renderer; the handler MUST NOT fake continuity across a gap. The renderer's reset-on-window-lost path is the observable behaviour in either case.
- **Closed-session replay is a separate frontend surface**, not a re-skin of the live terminal. The replay-only viewer mounts the standard `XtermRenderer` against a replay-only `TerminalSessionClient` shape; no `Input`, no `Resize`, no `Detach`/`Close` controls. The wire close after `ReplayEnd` is the signal that the recording has been fully delivered.
- **Backend-restart reconciliation is idempotent and audited.** On startup, the orchestrator scans `terminal_sessions WHERE status IN ('starting', 'active', 'detached')`, transitions each to `closed`, and inside the same database transaction appends one matching `closed` `session_events` row per session with payload `{ "reason": "startup_reconciliation", "previous_status": <starting|active|detached>, "reconciled_at": <UTC timestamp> }` AND — for any reconciled session that has at least one chunk row AND no equivalent existing marker — one `terminal_recording_markers { kind: closed, seq: MAX(seq_end), payload: { reason: "startup_reconciliation", "previous_status": <prior>, "reconciled_at": <UTC> } }` row. The marker payload is built field-by-field from public metadata only and never carries chunk bytes, `client_info`, peer banners, or error text. The pass writes ZERO `audit_events` rows on either branch (operational bookkeeping, matches the existing close-path audit shape) and never deletes any session, chunk, or marker row — recordings remain readable through the closed-session replay path. Reconciliation runs BEFORE the HTTP listener binds; a database failure on the reconciliation transaction fails the boot rather than starting a backend that would serve stale "live" rows the UI cannot resume. Idempotent on three axes: a second startup that finds no pre-restart non-`closed` rows is a byte-identical no-op (no events appended, no marker written, no rows touched); the marker insert uses `ON CONFLICT DO NOTHING` against the partial unique index `terminal_recording_markers_session_closed_seq_uidx` on `(terminal_session_id, seq) WHERE kind = 'closed'`, so a partial earlier run, an operator-written marker, or a racing writer all collapse to a single row at the schema layer; sessions with zero chunks never reach the marker branch. Live PTY recovery across the restart is intentionally NOT in scope.
- **At-rest encryption is opt-in via config in v1**, with the documented at-rest risk spelled out for the operator. The envelope shape mirrors the existing vault — magic prefix + version byte + 24-byte XChaCha20-Poly1305 nonce — but uses a SEPARATE master key. The `encryption` column is per-row so a future config flip does not have to re-encrypt history in-place.
- **Conservative retention defaults when enabled.** The future cleanup worker defaults to per-session 30-day retention from `closed_at` and a per-session 64 MiB byte cap (the writer emits a `replay_gap { reason: "byte_cap_reached" }` marker when adding the next chunk would exceed the cap). Retention sweeps delete chunk and marker rows only — `terminal_sessions` rows survive the sweep, matching the existing "`terminal_sessions` are NEVER deleted from the user UI" policy in "Inventory lifecycle and destructive-action policy".
- **Retention design landed; audit kind + repository purge primitive landed; typed `[terminal_recording.cleanup]` config foundation landed; Stage A startup-only sweep landed; Stage B periodic managed worker has now landed (`relayterm-terminal::retention_worker`, wired from `apps/backend/src/main.rs` after `AppState` construction and BEFORE the listener accepts connections, gated on `cleanup.enabled` AND `cleanup.periodic_sweep_enabled` AND `cleanup.sweep_interval_seconds > 0`, advisory-locked via `relayterm-db::PgRetentionAdvisoryLock` and shut down through a shared `tokio::sync::watch::Receiver<bool>`).** `docs/terminal-recording.md` Section 12 is the binding contract for the retention slice — eligibility (`closed_at + retention_days <= NOW()`, inclusive), what is preserved vs deleted (chunks and markers DELETE; `terminal_sessions`, `session_events`, `audit_events` KEEP), the single-transaction shape (`SELECT` aggregates → `DELETE markers` → `DELETE chunks` → `INSERT audit_events`; audit failure ROLLBACK reverts the deletes), the `recording_purged` audit row (kind, `actor_id = NULL` system actor, public-only payload of `target_id`/`target_kind`/`chunk_count`/`marker_count`/`bytes_purged`/`retention_days`/`closed_at`/`purged_at`/`reason: "retention_expired"`), the new `[terminal_recording.cleanup]` config block (`enabled`, `startup_sweep_enabled`, `sweep_interval_seconds` — sentinel `0` disables the periodic worker, otherwise `60..=604800`, `batch_size`) independent of `terminal_recording.enabled` so a recording corpus does NOT become immortal when recording is later disabled, the staged worker (Stage A startup-only sweep before the listener binds; Stage B periodic managed task with `pg_try_advisory_lock` and graceful-shutdown wiring — never `tokio::spawn`-and-forget), the post-purge API/UI shape (metadata returns `has_recording = false` byte-identical to a never-recorded session; bytes_purged comes from `SUM(byte_len)`; static log category tags only; deletion by `terminal_session_id` only, never caller-supplied id lists), and the test list (Section 12.12). The audit kind landed via migration `20260503000021_audit_events_recording_purged_kind.sql` + `AuditEventKind::RecordingPurged` (`relayterm-core::audit_event`); the single-session purge primitive landed as `TerminalRecordingRepository::purge_for_retention(input) -> Result<Option<PurgedRecordingSummary>>` with the input carrying `terminal_session_id`, `retention_days`, and the worker's authoritative `now`. The Postgres impl runs ONE transaction per session that locks the session row `FOR UPDATE`, evaluates the eligibility predicate at the inclusive boundary, aggregates `COUNT(*)` and `COALESCE(SUM(byte_len), 0)` from chunks plus `COUNT(*)` from markers (NEVER reads `payload` bytes), deletes markers then chunks, and inserts the `recording_purged` audit row (`actor_id = NULL`, payload built field-by-field from primitives) — all inside the same `BEGIN`...`COMMIT`. Audit failure ROLLBACK reverts the deletes (12.4 fail-closed). `Ok(None)` for unknown / open / in-window / zero-recording sessions writes no audit row; the schema-side idempotency keystone (predicate (3): `chunk_count > 0 OR marker_count > 0`) collapses a second purge into a byte-identical no-op. The `recording_purged` audit row's payload is invisible to the user-facing `recent_for_actor` feed by construction (`actor_id IS NULL`, the SQL filter is `WHERE actor_id = $caller`). Sentinel-tested redaction in `crates/relayterm-db/tests/repositories.rs::purge_for_retention_*` covers the chunk byte sentinel, the marker payload sentinel, the standard `AUDIT_FORBIDDEN_SUBSTRINGS` set, and `PurgedRecordingSummary` `Debug`. The typed `[terminal_recording.cleanup]` config block has now landed under `apps/backend/src/config.rs` — five fields (`enabled` default `true`, `startup_sweep_enabled` default `true`, `periodic_sweep_enabled` default `false`, `sweep_interval_seconds` default `0` sentinel = no periodic schedule, `batch_size` default `100`) plus their `RELAYTERM_TERMINAL_RECORDING__CLEANUP__*` env mirrors, validated against `batch_size 1..=10_000` and `sweep_interval_seconds` `0` OR `60..=604800`, with `periodic_sweep_enabled = true` requiring a non-zero in-bounds cadence. Validation runs INDEPENDENTLY of `terminal_recording.enabled` so a future worker may sweep an existing corpus even after recording is disabled (12.6 independence rule). Malformed env values fail boot with the variable name in the error rather than silently falling through. The fields are accepted at boot and otherwise unused — no caller consumes them yet, and the slice changes ZERO runtime behaviour. The Stage A startup retention sweep AND the Stage B periodic managed worker have now both landed (the worker is `relayterm-terminal::retention_worker::spawn_recording_retention_periodic_worker`, wired from `apps/backend/src/main.rs` between `AppState` construction and the listener bind, gated by `cleanup.enabled` AND `cleanup.periodic_sweep_enabled` AND a non-zero `cleanup.sweep_interval_seconds`, advisory-locked via `relayterm-db::PgRetentionAdvisoryLock`, shut down through a shared `tokio::sync::watch` channel, fail-soft on every error path, and using the same `run_recording_retention_sweep` body Stage A uses — cadence-tagged `SweepCadence::Periodic`).
- **Operator-facing warnings are pinned in tests.** The future enable-recording UI MUST display a static "this captures everything your terminal prints" warning; the future export UI MUST display a second static "exports may contain secrets the shell echoed" warning. Both strings are pinned in tests the same way the auth-check status strings are pinned today.
